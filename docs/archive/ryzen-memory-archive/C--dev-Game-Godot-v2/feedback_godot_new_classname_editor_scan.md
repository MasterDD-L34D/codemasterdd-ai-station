When you ADD a new `class_name` to a GDScript file, a headless GUT run via `godot --headless -s addons/gut/gut_cmdln.gd -gtest=...` fails to resolve the new type ("Could not find type X" / "Identifier X not declared", often with many cascading failures) because the new global is not yet in `.godot/global_script_class_cache.cfg`.

**Why:** the cache is regenerated by the editor's filesystem scan, not by a `-s` script run. A fresh `class_name` is invisible to the headless test runner until that happens.

**How to apply:** after creating a new `class_name` file (and before/around the first GUT run), regenerate the cache once:
```
<godot> --headless --editor --quit-after 2
```
Then the `-s` GUT run resolves the class. Do NOT stage `.godot/global_script_class_cache.cfg` (gitignored build artifact), but DO commit the script's `.uid` sidecar (`scripts/.../foo.gd.uid`) so a clean checkout / CI resolves the global class.

**Evidence (2026-05-21, M1 sub-proj 2):** recurred twice in one session — adding `SistemaApi` (PR-B1) and `MainSistema` (PR-C2). Both produced spurious parse-error cascades on the first headless run; the editor scan fixed both. Cost a confused minute each time until the pattern was recognized. CI must run a Godot editor import/scan pass after a new-class_name PR lands (or the committed `.uid` covers resolution).

**Godot binary on PC VGit (Ryzen):** `/c/Users/VGit/AppData/Local/Godot/Godot_v4.6.2-stable_win64_console.exe` (console exe for stdout; not in PATH). GUT pattern: `... --headless -s addons/gut/gut_cmdln.gd -gtest=res://tests/unit/<f>.gd -gexit`; full suite `-gdir=res://tests/unit`.
