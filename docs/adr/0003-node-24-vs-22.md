# ADR 0003 — Node.js: versione 24 invece di 22

**Status**: Accepted
**Data**: 2026-04-20
**Decisore**: Eduardo Scarpelli
**Tipo decisione**: tecnica (runtime, compatibilità)

## Contesto

### Situazione iniziale

Durante setup dev stack sul Lenovo (20 aprile 2026), ho pianificato
installazione **Node.js 22 LTS** come da CLAUDE.md iniziale.

Motivazione originale:
- Evo-Tactics (mio progetto principale) richiede `engines.node: ^22`
- Node 22 era considerato "LTS stabile" al momento della scrittura del CLAUDE.md
- Conservatismo per evitare sorprese

### Il risultato imprevisto

Comando eseguito:
```powershell
winget install OpenJS.NodeJS.LTS --accept-package-agreements --accept-source-agreements
```

**Winget ha installato Node.js 24.15.0**, non Node 22.

### Perché?

Il manifest winget `OpenJS.NodeJS.LTS` **punta sempre alla Active LTS corrente**.

**Node.js LTS timeline**:
- Node 20 "Iron": Active LTS dal 24/10/2023 → Maintenance da 28/10/2025
- Node 22 "Jod": Active LTS dal 29/10/2024 → Maintenance da 28/10/2025
- **Node 24 "Krypton": Active LTS dal 28/10/2025** (in corso)
- Node 26 "next" attesa release ottobre 2026

Quindi dal 28/10/2025 in poi, `winget install OpenJS.NodeJS.LTS` = Node 24.
Node 22 è ancora supportato (maintenance fino aprile 2027), ma non è più "latest LTS".

### La decisione richiesta

Tre opzioni:

**A)** Tenere Node 24 (raccomandato da Claude)
**B)** Downgrade esplicito a Node 22 (winget install --version 22.x.x)
**C)** Installare nvm-windows per gestire versioni multiple

## Decisione

Scelgo **Opzione A**: tengo **Node.js 24.15.0** come versione di sistema.

### Ragionamento dettagliato

**Ragione 1: supporto lungo**
- Node 24 Active LTS fino ad aprile 2029 = 3 anni sicuri
- Node 22 Maintenance fino aprile 2027 = 1 anno sicuro

**Ragione 2: compatibilità Synesthesia**
- Synesthesia (mio progetto esame) è esplicitamente "testato Node 24"
- Quindi Node 24 è già validato per almeno un mio progetto

**Ragione 3: compatibilità Evo-Tactics probabile**
- Evo-Tactics usa `engines.node: ^22` (caret, non strict)
- `^22` significa "22.x, 23.x, 24.x, ..." (compatibile semver-wise con 24)
- **Al peggio Node 24 emette warning** "engine mismatch", ma funziona

**Ragione 4: future-proof**
- Progetti nuovi avranno engines.node più alti
- Node 24 è punto di partenza future-proof

**Ragione 5: ecosystem maturity**
- A 6 mesi da release Node 24, native modules (better-sqlite3, node-sass, ecc.)
  hanno pre-built binaries per Node 24
- npm 11.12.1 supporta features moderne (pacchetti, workspace, ecc.)

**Ragione 6: YAGNI per nvm-windows**
- Installare ora nvm-windows è pre-emptive
- Non ho ancora visto conflitti reali
- Se/quando emergono, installo in 5 minuti

### Quando scatterà Opzione C (nvm-windows)

**Trigger specifici** per installare nvm-windows:
1. Evo-Tactics ha `engines.node: "~22.x"` (strict) invece di `^22` → dovrò usare 22
2. Altri progetti miei iniziano con engines.node diversi tra loro
3. Freelance o collaborazione richiede Node specifico diverso

**Trigger NON validi**:
- "Potrebbe servire in futuro" (YAGNI)
- "I dev senior hanno sempre nvm" (cargo culting)
- "Per completezza" (non giustifica complessità)

## Conseguenze

### Positive

**Supporto lungo garantito**:
- 3 anni di security updates + feature releases

**Performance migliori**:
- Node 24 ha miglioramenti V8 (importabili attributes native, built-in test runner più maturo, performance improvements)
- Sync fs ops più veloci, ReadableStream miglioramenti

**Compatibilità ampia**:
- Synesthesia validato
- Evo-Tactics `^22` compatibile
- Nuovi progetti beneficiano

**Semplicità operativa**:
- Un Node installato, no gestione versioni
- Less complexity = less bug surface

### Negative

**Possibili warning su Evo-Tactics**:
- Se `engines.node: ^22` è "^22.0.0", Node 24 rientra
- Se `"^22.x"` (notazione più rigida), potrebbe emettere warning
- Non errore, solo warning (pacchetto girerà comunque)

**Fix-forward mindset obbligato**:
- Se un progetto rompe su Node 24, devo fix (upgrade dependencies)
  invece di usare Node 22

**Dipendenza da winget**:
- Se winget cambia manifest LTS in futuro, dovrò essere consapevole
- Mitigation: pinnare versione (`--version 24.15.0`) in script reproducibili

### Mitigation per rischi negativi

**Rischio 1**: Evo-Tactics emette warning
**Mitigation**: accettabile, comunque funziona. Se fastidioso, update
engines.node a "^22 || ^24".

**Rischio 2**: Native module non compila per Node 24
**Mitigation**: rebuild con `npm rebuild`, se fallisce cerco alternative package
o installo nvm-windows per quel progetto specifico.

**Rischio 3**: Un progetto futuro richiede Node 20 rigido
**Mitigation**: installo nvm-windows quando succede (non prima).

## Alternative considerate

### Alternative B: Downgrade esplicito a Node 22

**Comando**: `winget uninstall OpenJS.NodeJS.LTS; winget install OpenJS.NodeJS --version 22.x.x`

**Pro**:
- Allineamento esatto con CLAUDE.md originale
- Zero warning su Evo-Tactics

**Contro**:
- Workflow più complesso
- Supporto più breve (2 anni)
- Future-proof ridotto

**Perché scartata**:
- Le ragioni di Node 24 (supporto 3y, validazione Synesthesia, future-proof)
  superano le ragioni di Node 22 (allineamento a CLAUDE.md scritto "generico")
- CLAUDE.md è documento vivo: si aggiorna, non è contratto

### Alternative C: nvm-windows subito

**Pro**:
- Flessibilità totale su versioni multiple
- Workflow "standard" per dev multi-progetto

**Contro**:
- Complessità aggiuntiva non ancora giustificata
- YAGNI: non ho progetti che la richiedono oggi
- Più setup + learning curve

**Perché scartata**: YAGNI. Valida se problemi emergono.

### Alternative D: nodenv o volta (alternative a nvm-windows)

**Pro**:
- volta è molto veloce, .node-version friendly
- nodenv è cross-platform-minded

**Contro**:
- Meno diffusi di nvm su Windows
- Additional tools senza necessità

**Perché scartata**: anche questi YAGNI.

## Procedure

### Verifica Node 24 dopo install
```powershell
node --version    # atteso: v24.15.0
npm --version     # atteso: 11.x.x
```

### Test rapido funzionalità Node 24 features
```powershell
# Test import attributes (Node 24 native)
node --input-type=module -e "const pkg = await import('./package.json', { with: { type: 'json' } })"

# Test built-in test runner
node --test --experimental-test-coverage <testfile>
```

### Se Evo-Tactics emette warning su Node 24

Verifica output:
```powershell
cd C:\dev\Game
npm install
```

Se warning simile a:
```
npm WARN EBADENGINE Unsupported engine {
  package: 'evo-tactics@0.x.x',
  required: { node: '^22' },
  current: { node: 'v24.15.0' }
}
```

**Opzioni**:
1. **Ignora** (warning, non errore)
2. **Update** engines.node in package.json Evo-Tactics:
   ```json
   {
     "engines": {
       "node": "^22 || ^24"
     }
   }
   ```
3. **Install nvm-windows** come escalation (solo se veramente problematico)

### Se un native module non compila

```powershell
cd <project-dir>
npm rebuild
```

Se fallisce:
```powershell
rm -rf node_modules
rm package-lock.json
npm install
```

Se ancora fallisce, investigate package specifico. A volte serve
`npm install --build-from-source <package>`.

## Meta-learning

### Cosa ho imparato

**1. Leggere output comando prima di "prossimo"**
Se avessi chiuso gli occhi su "Node 24.15.0" nell'output, avrei avuto un
disallineamento CLAUDE.md silente. **Leggere ogni output è sempre worth 30 secondi**.

**2. Il manifest winget ha logica "latest stable"**
Non è bug: `OpenJS.NodeJS.LTS` è per definizione "current Active LTS".
Se voglio pinning, devo essere esplicito: `--version 22.x.x` o `--version 24.15.0`.

**3. Pragmatismo > coerenza ideologica**
CLAUDE.md diceva Node 22. Realtà ha fornito Node 24. Invece di forzare
downgrade "per coerenza", ho valutato vero best choice e aggiornato CLAUDE.md.
**Il documento serve me, non viceversa**.

**4. Semver caret `^` è friend**
`^22` permette 22.x, 23.x, 24.x, ..., fino a 30 se esiste. Questo **è by design**
per consentire bug fix e minor updates. Node 24 è "minor" in semver terms
(non breaking per la stragrande maggioranza dei casi).

**5. YAGNI vs FOMO**
Installare nvm-windows "per sicurezza" = FOMO (fear of missing out).
Installarlo quando serve = YAGNI rispettato.

**La differenza sta in: ho un problema reale da risolvere?**
- "Potrei avere" ≠ problema reale
- "Ho warning che non tollero" = problema reale

| 20      | Iron    | Apr 2023  | Oct 2023   | Oct 2025    | Apr 2026    |

"Maintenance" = solo security patches critici.
**End-of-Life** = supporto terminato, upgrade obbligato.
